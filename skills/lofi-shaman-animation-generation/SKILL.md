## Phase 1: Animation — Build the Loop First, Preview Before Rendering Full Length

> ⚠️ **Always show Aidan a short preview (3–25 seconds) before rendering the full-length video.**

### Setup
```bash
pip3 install moviepy --break-system-packages
```
MoviePy v2+ changed imports — use `from moviepy import VideoClip` not `from moviepy.editor import VideoClip`.

### Render at 1080p, not 2K
2K (2752×1536) frames are too slow for MoviePy (~1.3 fps). Always downscale the background on load:
```python
W, H = 1920, 1072  # 1080p, divisible by 16 for encoder
bg_img = Image.open("layers/01_background_no_ball.png").convert("RGB").resize((W, H), Image.LANCZOS)
```
At 1080p, render speed is ~3 fps — a 20-second loop takes ~2.5 minutes.

### Render & export
```python
from moviepy import VideoClip

clip = VideoClip(make_frame, duration=DURATION)
clip.write_videofile(
    "output.mp4",
    fps=24,
    codec="libx264",
    audio=False,
    preset="fast",
    ffmpeg_params=["-crf", "22", "-pix_fmt", "yuv420p", "-movflags", "+faststart"]
)
```

### Render strategy — short base loop, then FFmpeg loop
**Do NOT render the full 1-4 hour video with MoviePy.** Render a short seamless base loop and use FFmpeg to multiply it.

The loop duration must be chosen so that all animations return to their starting state:
- Ripples: seamless at any duration (sine wave)
- Action every 20s → **use a 20-second base loop**

### Lessons learned — Phase 1
- Always test with a 3-second render before a full 25-second preview
- MoviePy sessions get killed by the OpenClaw exec timeout at ~2K resolution — always use 1080p
- The `SIGTERM` issue: long ffmpeg re-encode sessions get killed. Run as a **detached background process** using `&` and monitor via `tail -f /tmp/ffmpeg_mux.log`
- Ball coordinates: DO NOT use automated color detection. Use the coordinate picker tool on Vercel. Orange autumn leaves will confuse any color-based detection.
- The ball sprite generated by Nano Banana will be full-frame size — always resize to scene-appropriate diameter (50–70px for a midground character)

## Phase 2: Final Assembly — FFmpeg Mux Audio + Looped Video

This is the final step: loop the base video to match the audio length and merge them.

### Prerequisites
- Know the loop count: `floor(audio_duration / loop_duration)` = `floor(1040 / 20)` = 52 loops → `-stream_loop 51` (0-indexed)

**Key flags explained:**
- `-stream_loop 51` — loops the input video 52 times total (original + 51 repeats)
- `-map 0:v -map 1:a` — video from input 0, audio from input 1
- `-c:v libx264 -b:v 350k` — re-encode video at 350kbps (gives ~61MB for 17:20)
- `-shortest` — stop when the shorter stream (audio) ends
- `-movflags +faststart` — move moov atom to front for browser streaming
- `> /tmp/ffmpeg_mux.log 2>&1 &` — **run in background** to avoid session timeout killing it

### Monitor progress
```bash
tail -f /tmp/ffmpeg_mux.log
# Or just check periodically:
tail -2 /tmp/ffmpeg_mux.log
ls -lh lofi_shaman_final.mp4
```

### Verify completion
```bash
ffprobe lofi_shaman_final.mp4 2>&1 | grep Duration
# Should show full audio duration (17:20)
```

### Deliverables
1) The short,rendered video to repeat on loop
  - Send to Aidan on Telegram for sign off
2) Full video file - include file location in telegram chat
  - Duration should matches the master audio time

### Lessons learned — Phase 2
- **Never use `-c:v copy`** for the final mux when combining with stream_loop — the output will be ~134MB (too large for Vercel). Always re-encode at a target bitrate.
- **The moov atom issue:** with `-movflags +faststart`, ffmpeg does a two-pass write. The file will appear invalid (`moov atom not found`) until the process fully completes. Don't panic — just wait.
- **Session timeout kills ffmpeg** — always run as a background process (`&`) with output redirected to a log file. This is the only reliable way to complete a 17-minute encode.
- **`-shortest` is essential** — without it, the video will run past the audio end (silence for the remaining loops)
- The audio `master_audio.m4a` is already pre-built from the Suno workflow — the video loop just needs to match its duration

## Challenges Encountered & What To Do Better Next Time

### 1. Automated ball coordinate detection failed completely
**What happened:** Tried to find the basketball position in the scene using color detection (orange pixel clusters). The autumn leaves in the scene were the same orange as the ball — every detection attempt found the wrong region.

**Next time:** Skip color detection entirely for illustrated scenes. Go straight to the coordinate picker tool (`/coords.html` on Vercel). Have Aidan click the object directly. Takes 30 seconds and is always right.

---

### 2. 2K rendering was too slow and got killed
**What happened:** Tried rendering frames at 2752×1536. MoviePy hit ~1.3 fps, the OpenClaw exec session timed out with SIGTERM, and the process was killed before producing any output.

**Next time:** Always downscale the background to 1080p (1920×1072) at load time before animating. Rendering runs at ~3 fps at 1080p — fast enough. The source 2K image still provides crisp detail after downscale.

---

### 3. Long FFmpeg encodes got killed by session timeouts
**What happened:** Running FFmpeg to mux a 17-minute video synchronously inside an exec session hit the timeout and was SIGTERMed mid-encode — multiple times. The partially-written file appeared valid (ffprobe showed a duration) but was actually truncated.

**Next time:** Always run long FFmpeg jobs as a detached background process:
```bash
ffmpeg [args] > /tmp/ffmpeg_mux.log 2>&1 &
echo "PID: $!"
```
Monitor with `tail -2 /tmp/ffmpeg_mux.log` and `ps -p <PID>`. Don't try to wait inline.

---

### 4. Used `-c:v copy` which blew up the file size
**What happened:** First mux attempt used `-c:v copy` (stream copy, no re-encode). The base loop was encoded at ~1090kbps. A 17:20 video at that bitrate = 134MB — too large for Vercel edge caching, and the encode still got killed.

**Next time:** Always re-encode with `-c:v libx264 -b:v 350k`. Output is ~61MB at 17:20, streams cleanly from Vercel. See bitrate table in Phase 4.

---

### 5. Nano Banana doesn't actually produce transparent PNGs
**What happened:** Asked Nano Banana for a ball sprite with a "transparent background." The output was a full-frame 2K RGB image with the ball on a dark gray (~52,52,52) background — no alpha channel at all.

**Next time:** Always apply a circular mask in code to fake the alpha. Don't expect any AI image generator to produce a truly transparent PNG. The sprite extraction code in Phase 2 handles this correctly — use it.

---

### 6. No concept of "base loop first" — tried to render full length directly
**What happened:** Initially attempted to render the full 17:20 video frame-by-frame with MoviePy. This would have taken hours even at 1080p, and would have been killed by timeouts anyway.

**Next time:** Always think in loops. Identify the shortest repeating cycle (dribble period = 20s), render just that loop (~2.5 min), then use FFmpeg to tile it to the audio length. This is orders of magnitude faster and produces the same result.

---

### Things that worked well (keep doing these)
- **Coordinate picker tool** — building a lightweight Vercel-hosted tool for Aidan to click and copy coordinates was the right call. Reuse this pattern for any future positioning tasks.
- **20-second base loop + FFmpeg tile** — perfect strategy. Easy to re-render if something is wrong, fast to mux.
- **Background process pattern** — once committed to `&` + log file for FFmpeg, everything worked first try.
- **Ripple animation design** — sine wave formula is inherently seamless, no special loop handling needed. Use this math for any future water/ambient animation.
- **Approval gates** — showing Aidan a 3-second preview before a 25-second one before the full render saved a lot of wasted time from earlier sessions.
